# Experiment: Codebase Map Injection

## Status: READY TO RUN

## Question

Does injecting a pre-built codebase map into each sprint's context reduce token waste by preventing architectural mistakes and eliminating exploration overhead?

## Background

The v1.2 factorial experiment tested two instruction-level mutations (lint pre-check, cross-project lessons) across 5 products. Results:

- **Lessons (B)**: small, inconsistent efficiency gain (-3 tok/loc avg, inside noise floor)
- **Lint pre-check (C)**: net negative (+10 tok/loc avg, overhead exceeds benefit)
- **Combined (D)**: actively harmful (+18 tok/loc interaction, interference not synergy)
- **Noise floor**: 10-13% between replicates on token efficiency

Instruction-level mutations don't move the needle. The agent's efficiency is dominated by phase difficulty and architectural decisions, not by how well it follows process instructions.

This experiment tests a different mechanism: **structural knowledge**. When an agent doesn't understand the existing codebase, it writes code that conflicts with existing patterns, then rewrites it. A codebase map eliminates that class of error. This is not "better instructions" — it's "better tools."

## Design: Matched-Pair, 2 Products

Two products, each built twice: once with flat context (control), once with codebase map injection (treatment). 4 builds total, ~80 sprints.

|  | Product 1 (notegrep, TS) | Product 2 (pollster, Python) |
|--|--------------------------|------------------------------|
| **Control** | Flat context, condition-a SKILL.md | Flat context, condition-a SKILL.md |
| **Map** | Codebase map injected from sprint 2 onward | Codebase map injected from sprint 2 onward |

### Why 2 products, not 8

The v1.2 experiment showed that N=8 small products is expensive and the per-build noise is high. This experiment trades breadth for depth: 2 products with ~10 sprints each (longer roadmaps) instead of 8 products with 5 sprints. Longer projects give the treatment more time to compound and the learning curve metric more data points per build.

### Why notegrep and pollster

- **notegrep** (TypeScript/Node): markdown indexing, full-text search, TUI. Multiple interacting subsystems (indexer, search engine, UI layer) where structural knowledge matters.
- **pollster** (Python/FastAPI): REST API + CLI + SQLite. Layered architecture (routes, services, models, CLI) where the agent needs to understand the existing call graph to extend it correctly.

Both are from the v1.2 product roster but were not built during the factorial experiment (notegrep and pollster waves were never launched), so they're uncontaminated.

### Why not reuse fconv/presskit/pipesmith

Those products are already built under multiple conditions. Using them would introduce carry-over knowledge — the model's training data may now include patterns from the v1.2 builds. Fresh products avoid this.

## The Treatment: Codebase Map

After each sprint's code is committed (end of Phase 3), a map is generated and saved. The next sprint's subagent prompt includes this map.

### Map format

```
# Codebase Map (auto-generated after sprint N)

## File Tree
src/
  indexer.ts        — Markdown file scanner, frontmatter extraction, index builder
  search.ts         — Full-text search engine, fuzzy matching, result ranking
  tui.ts            — Terminal UI, keyboard navigation, result display
  types.ts          — Shared type definitions (Document, SearchResult, Config)
  config.ts         — Config loading from YAML, defaults, validation
tests/
  indexer.test.ts   — 23 tests: scanning, frontmatter parsing, incremental updates
  search.test.ts    — 18 tests: exact match, fuzzy, ranking, empty corpus
  tui.test.ts       — 12 tests: rendering, key handling, scroll state

## Module Dependencies
indexer → types, config
search → types, indexer (reads index)
tui → types, search (queries), config
cli → config, indexer, search, tui

## Key Patterns
- All async functions return Promise<T>, no callbacks
- Errors are typed: IndexError, SearchError, ConfigError (all extend BaseError)
- Tests use vitest with in-memory fixtures, no filesystem access
- Config is loaded once at startup and passed as parameter (no globals)

## Recent Changes (sprint N)
- Added fuzzy matching to search.ts (Levenshtein distance)
- Refactored indexer to support incremental updates (hash-based change detection)
- 53 tests total, 96% coverage
```

### Map generation

The map is generated by a script (`mapgen.sh`) that runs at the end of each sprint:

1. **File tree**: `find src/ -name "*.ts" -o -name "*.py"` etc., with one-line descriptions extracted from the first docstring/comment in each file
2. **Module dependencies**: parse import statements to build a dependency graph
3. **Key patterns**: extracted from the project's CLAUDE.md conventions section + inferred from recurring patterns in the codebase (error handling style, test patterns, config approach)
4. **Recent changes**: `git diff --stat HEAD~1` + summary of what changed

The map is capped at 80 lines. If the codebase grows beyond what fits in 80 lines, the map shows only source files (drops test file descriptions) and collapses stable modules into one-liners.

### Where the map goes

Saved to `{FLOWSTATE}/codebase-map.md`. Added to the subagent prompt in Auto-Continue (step 6), between the progress.md read and the "Run Sprint N+1" instruction:

```
- "Read `{FLOWSTATE}/codebase-map.md` for the current codebase structure, module dependencies, and key patterns. Use this to understand where new code should go and how it should integrate with existing modules. Do not re-explore files that the map already describes."
```

### What the map does NOT do

- Does not replace reading actual source files during implementation
- Does not contain code snippets (that would bloat the context)
- Does not include generated files, config files, or lock files
- Does not persist across experiments (each build generates its own)

## Map Generation Script

The map is generated by `mapgen.sh`, invoked at the end of Phase 3 after the sprint commit. It is stack-aware:

```bash
#!/usr/bin/env bash
# mapgen.sh — Generate codebase map for the next sprint
# Usage: ./mapgen.sh <project-dir> <flowstate-dir> <sprint-number>

PROJECT_DIR="$1"
FLOWSTATE_DIR="$2"
SPRINT_NUM="$3"
OUTPUT="$FLOWSTATE_DIR/codebase-map.md"
MAX_LINES=80
```

For each supported stack:
- **TypeScript**: parse `import ... from` statements, extract first `/** ... */` or `// ...` comment per file
- **Python**: parse `from X import` / `import X`, extract module docstrings
- **Go**: parse `import (...)` blocks, extract package-level comments
- **Rust**: parse `use` statements, extract `//!` module docs

The script is deterministic (same codebase state = same map) so it doesn't add noise between replicates.

## Control Condition

The control uses condition-a.md (v1.1 baseline) unmodified. No map is generated, no map is injected. The subagent discovers the codebase structure by reading files during Phase 1, as it does today.

## Treatment Condition

The treatment uses condition-a.md with two additions:

1. **Phase 3 addition** (after step 5, commit): "Run `mapgen.sh` to generate the codebase map at `{FLOWSTATE}/codebase-map.md`."

2. **Auto-Continue addition** (step 6, subagent prompt): Include codebase-map.md in the context loading list, with the instruction above.

No other changes. The treatment isolates the map — no lessons, no lint pre-check, no other v1.2 features.

## Extended Roadmaps

Both products get extended PRDs with 10 phases instead of 5, to give the learning curve more data points. The extensions add realistic scope:

**notegrep (10 phases):**
1. Project scaffold and content model
2. Markdown scanner and frontmatter extraction
3. Full-text search index (inverted index)
4. Fuzzy search and result ranking
5. CLI interface (search, list, show)
6. TUI browser (scrollable results, preview pane)
7. Tag-based filtering and boolean queries
8. Incremental index updates (file watching)
9. Search result export (JSON, markdown)
10. Performance optimization (large corpus handling)

**pollster (10 phases):**
1. Project scaffold and data model
2. SQLite storage and migration
3. REST API: create polls, cast votes
4. Vote deduplication and validation
5. Results aggregation and ranking
6. CLI client (create, vote, results)
7. Real-time results (SSE endpoint)
8. Poll templates and bulk creation
9. Export/import (JSON, CSV)
10. Rate limiting and abuse prevention

Each phase is scoped to 500-1500 LOC, consistent with v1.2 sizing.

## Metrics

### Primary: Learning Curve Slope

For each build, compute tokens-per-LOC at each sprint (excluding S0). Fit a linear regression. The slope is the learning rate:
- **Negative slope** = agent gets more efficient over time (learning)
- **Zero slope** = no learning
- **Positive slope** = agent gets worse (context degradation)

Compare slopes: treatment vs control, within each product.

### Secondary

1. **Rework rate per sprint** — does the agent rewrite less code as the project matures? Lower rework in map builds suggests the map prevents architectural mistakes.
2. **Context compressions per sprint** — does better context loading reduce the need for compressions? Fewer compressions in map builds suggests the map saves context budget.
3. **Phase 1 token usage** — how many tokens does the agent spend in THINK (orientation + planning) vs EXECUTE? The map should reduce THINK tokens.
4. **Token efficiency (aggregate)** — same as v1.2: total new_work_tokens / total loc_added per build.

### Not a metric

Gates-first-pass rate. The v1.2 experiment showed this is too noisy to be informative at any sample size we can afford.

## Analysis

### Per-product paired comparison

For each product:
- Control slope vs treatment slope (the primary comparison)
- Per-sprint token efficiency trajectories (visual: do the lines diverge over time?)
- Aggregate efficiency (treatment total tok/loc vs control total tok/loc)

### Effect size

Report the difference in slopes with a bootstrap CI. Also report the raw efficiency difference as a percentage, for comparability with v1.2 results.

### Noise floor reference

Use the v1.2 replicate data (fconv A vs A2, presskit A vs A2) as the noise floor benchmark. If the map effect is smaller than the replicate variance, it's indistinguishable from noise.

### What counts as a finding

- **Slope difference > 5 tok/loc per sprint**: meaningful effect. At 10 sprints, that's 50 tok/loc cumulative — well outside the v1.2 noise floor.
- **Slope difference 2-5 tok/loc per sprint**: suggestive but within noise. Worth noting, not worth building on.
- **Slope difference < 2 tok/loc per sprint**: null. The map doesn't help measurably.

## Execution

### Infrastructure

Same VPS (100.87.64.104), same launcher infrastructure as v1.2. 2 builds run concurrently (one per product, matched pair runs sequentially to avoid resource contention).

### Build order

1. notegrep-control + pollster-control (concurrent)
2. notegrep-map + pollster-map (concurrent)

Controls first so the map generation script can be validated on the control builds' codebase (dry run) before the treatment builds depend on it.

### Launch

```bash
./launcher.sh notegrep control
./launcher.sh pollster control
# wait for completion
./launcher.sh notegrep map
./launcher.sh pollster map
```

### Duration estimate

10 sprints x ~30 min each x 4 builds = ~20 hours of VPS time. With 2 concurrent builds per wave, ~10 hours wall time per wave, ~20 hours total.

## Prerequisites

1. Write extended PRDs for notegrep (10 phases) and pollster (10 phases)
2. Write `mapgen.sh` for TypeScript and Python stacks
3. Create condition-map.md (condition-a.md + map generation and injection)
4. Validate mapgen.sh against an existing completed build (e.g., fconv-a or pipesmith-b) to confirm it produces useful output
5. Update launcher.sh to accept "control" and "map" as condition names

## Relationship to v1.2

This experiment is independent of the v1.2 factorial. It does not test lessons or lint. It reuses:
- The launcher infrastructure
- The metric collection pipeline
- The noise floor data from v1.2 replicates
- Two unused products from the v1.2 roster

If the map shows a positive effect, it can be added to the Flowstate workflow as a Phase 3 step without interacting with any v1.2 feature.

## Limitations

- **N=2 products.** Same limitation as always. The effect may be product-specific.
- **No replicates.** Budget constraint. The v1.2 noise floor (10-13%) is used as a proxy.
- **Map quality depends on the script.** A bad map could hurt by injecting wrong information. The script must be validated before the experiment.
- **10 sprints may not be enough.** If the learning curve only bends at sprint 15+, 10 sprints won't capture it. But 10 is the maximum practical scope for this experiment.
- **Confounded with phase difficulty.** Later phases (performance optimization, abuse prevention) are inherently different from early phases (scaffold, CRUD). The matched-pair design controls for this (both builds face the same phases in the same order), but it means the slope measures "phase ordering + treatment" not "treatment alone."
