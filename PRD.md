# Flowstate

A sprint-based development workflow for Claude Code. Gates for quality, structure for scope control, metrics for tracking.

## What Flowstate Does

Flowstate gives you three things:

1. **Quality gates** -- automated checks (tests, type checker, linter, coverage) that run after every sprint. Non-negotiable pass/fail. The single most reliable mechanism in the system.

2. **Sprint structure** -- a three-phase workflow (Think, Execute, Ship) that prevents wrong turns on multi-module work. Optional for small tasks.

3. **Metrics collection** -- token usage, session time, test counts, coverage, LOC. Imported into `sprints.json` after each sprint for tracking trends.

## How It Works

### The Sprint Cycle

1. **Start**: Record a baseline (SHA, test count, coverage, gate status)
2. **Phase 1+2 (Think + Execute)**: Paste the sprint prompt. The agent plans, implements, and runs gates. One prompt, no human breaks.
3. **Phase 3 (Ship)**: Paste the ship prompt. The agent writes a retro, updates the roadmap, and prepares the import JSON.
4. **Import**: Run `/import` or `python3 tools/import_sprint.py --from <path>` to save metrics to `sprints.json`.

### Gates

Three mechanical checks, run before any code ships:

| Gate | What it checks | Example |
|------|---------------|---------|
| Tests pass | Full test suite, not just new tests | `npm test`, `cargo test`, `pytest` |
| No regressions | Test count >= baseline, coverage >= baseline | Compare against baseline file |
| Build succeeds | Clean compile, no type errors, lint clean | `tsc --noEmit`, `eslint .`, `cargo clippy` |

Gates don't require agent judgment. They run after code is written and give a binary answer: ship or fix.

### When to Use Structure vs. Skip It

- **Multi-module work** (15+ files, multiple subsystems): Use the full three-phase sprint. Planning prevents wrong turns.
- **Single-module work** (one feature, one area): Skip planning. Implement directly, run gates.

## File Layout

```
~/.flowstate/{project-slug}/
  metrics/
    baseline-sprint-N.md    # Starting state for each sprint
    sprint-N-import.json    # Generated by the agent in Phase 3
  retrospectives/
    sprint-N.md             # Agent's retro from Phase 3

your-project/
  .claude/skills/           # Planning skills (PM, UX, Architect)
  docs/ROADMAP.md           # Phases broken into sprint-sized chunks
```

## Skills

Flowstate includes five skill files in `skills/`. Based on testing across 19 sprints:

- **PM, UX, Architect**: Useful as thinking partners for roadmap planning between sprints. Copy these to `.claude/skills/` in your project.
- **Production Engineer, Security Auditor**: Specialized. Copy if your project needs them.

Skills don't measurably improve autonomous implementation quality (tested in Experiment 1). They're valuable for interactive planning conversations, not for telling the agent how to code.

## Sprint Templates

- `tier-1/sprint.md` -- Standard sprint template. Local machine, full tool access.
- `tier-2/sprint.md` -- Restricted environment (e.g., work laptop). No proprietary code leaves the machine.
- `tier-3/sprint.md` -- Minimal. Cloud IDE or limited access.

Most projects use Tier 1.

## Metrics

After each sprint, metrics are imported into `sprints.json`. Fields tracked:

- Active session time
- Token usage (total, new-work, by model)
- Tests (total, added), coverage
- LOC added
- Gate pass/fail
- Context compressions

Use the MCP server (`tools/mcp_server.py`) to collect metrics directly from Claude Code session logs, or `tier-1/collect.sh` for shell-based collection.

## Research Background

Flowstate was developed over 19 sprints across three projects (TypeScript, Rust, SvelteKit). The full research data -- experiment designs, hypothesis tracking, and raw sprint data -- is preserved on the [`archive/v1-research`](https://github.com/smledbetter/flowstate/tree/archive/v1-research) branch.

Key findings:
- Gates catch real bugs reliably (3/3 planted bugs caught in adversarial testing)
- Sprint structure prevents scope drift on multi-module work (held at 2.5x scope)
- Skills don't measurably improve implementation quality (76-88% blind review with/without)
- Single-agent outperforms multi-agent delegation on moderate codebases
- Retrospective-driven skill evolution stabilizes quickly (3-4 sprints)
